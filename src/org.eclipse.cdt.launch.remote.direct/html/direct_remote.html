Debugging Remote C/C++ Applications Directly

The Direct RSE Remote CDT Plugin builds on top of CDT and integrates with RSE to introduce a "Remote Debugging C/C++ Application Directly" launch configuration. This launch configuration relies on the shell services of the RSE connections.

The "Remote Debugging C/C++ Application Directly" launch configuration enables the user to select from a list of RSE connections which have the shell service available. The gdb is invoked in the remote host using the shell service. The debugging is running exactly like local debug but all gdb debug command are sent to the remote gdb instead of calling the local gdb

The local platform and remote platform can be different. For example local platform can be Windows and remote platform can be Linux. The exe file must in remote platform format and locates in the remote platform. User enters its paths in the settings

If you have two copies of the source files. One in local host which is included in Eclipse project and other in the remote host which you built the exe with, then you must put in "Remote workspace directly" the path of the source files in the remote host. The is necessarily to make the break points work fine

System Prerequisites

Local Host
  • It is necessary to create an RSE connection which has the shell services available.
Remote Host
  • gdb debugger installed

Using

Run Application

You need to build your application on remote host, then put the absolute path of the exe file in the settings. The debugger will run the exe file and debug it

Attach Application

You can check the attach check box to attach into a running application. The debugger will show "Select Process" dialog while debugging the application.

Remote workspace directory

If you have local and remote copies of the source code. You must specify in "Remote workspace directory" the folder in the remote directory which contains the source files. This is the corresponding folder of the local source files.
